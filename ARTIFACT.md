# STRIDE — Reproducibility Artifact

This document is a step-by-step recipe for reproducing the measurements in the
STRIDE paper (`docs/paper_visual_computer_v2.md`). Every table and every plotted
number is generated by a script in this repository. CSVs in the repo root
(`bench_*.csv`) are the exact outputs the paper cites.

If you only want to read or cite the numbers, see the paper's Tables 1–9. If
you want to verify them on your own hardware, follow the steps below.

---

## 1. Hardware and software

| Component | Reference setup | Notes |
|---|---|---|
| GPU         | NVIDIA RTX 3090 (Ampere SM 8.6, 24 GB) | Any CUDA-12 GPU with ≥ 8 GB VRAM works; throughput scales with SM count. |
| CPU         | Intel Core i9-12900K        | Single-threaded encode + CPU competitor timings. |
| OS          | Windows 11                  | Linux works for everything. |
| CUDA        | 12.x                        | Must match the installed `cupy-cuda12x` build. |
| Python      | 3.11                        | 3.10–3.12 also tested. |

DGF GPU throughput numbers in the paper (§5.3, Table 4) are *cited* from AMD's
public DGF benchmark page [9] on an RX 9070 XT — we do not re-time DGF on our
hardware. All other GPU numbers are measured here.

## 2. Install

```bash
git clone https://github.com/maletsden/meshpress.git
cd meshpress
python -m venv .venv && .venv\Scripts\activate    # Windows
# source .venv/bin/activate                       # Linux/macOS
pip install -r requirements.txt
```

`cupy-cuda12x` and `DracoPy` are the load-bearing third-party deps for the GPU
decode benchmarks (§5.3) and the Draco competitor column (§5.2). If `pip
install cupy-cuda12x` fails, see [the CuPy installation
guide](https://docs.cupy.dev/en/stable/install.html) for the build matching
your CUDA toolkit.

Optional: build the DGF and Corto reference encoders for competitor parity:

```bash
# DGF reference encoder (paper Table 3 'DGF tb12' column)
cd third_party/DGF-SDK && cmake -B build -S . && cmake --build build --config Release
# Corto encoder + bench (paper Table 3/4 'Corto v12' column)
cd ../corto && cmake -B build -S . && cmake --build build --config Release
```

Without these the corresponding competitor columns are simply skipped — the
STRIDE numbers reproduce regardless.

## 3. Datasets

The §5.1 corpus has eight meshes. Five are auto-fetchable from public
scientific repositories (Stanford 3D Scanning, Common 3D Test Models). The
remaining three (Blender Monkey at high-tessellation export, Threedscans Dark
Finger Reef Crab, the `tank` game asset) are provided as-is in `assets/`. If
you start from a clean checkout, the auto-fetchable five land in `assets/` via:

```bash
python scripts/download_models.py --paper
```

Expected `assets/` after the fetch + the provided meshes:

```
assets/
  fandisk.obj           # Common 3D Test Models (CC0)
  stanford-bunny.obj    # Stanford 3D Scanning Repository
  horse.obj             # Common 3D Test Models (CC0)
  Monkey.obj            # Blender Suzanne, exported with 6 levels of subdiv
  happy_buddha.obj      # Stanford 3D Scanning Repository
  crab.obj              # Threedscans "Dark Finger Reef Crab" (CC0)
  tank.obj              # Game asset (positions + indices only)
  xyzrgb_dragon.obj     # Stanford 3D Scanning Repository (xyzrgb set)
```

Vertex / triangle counts after dedup (the encoder's pre-pass) should match
paper Table 2.

## 4. Reproduction recipes

Every script writes its CSV to the repo root, overwriting any existing file
of the same name. Re-running is idempotent.

### Table 1 — Meshlet generator comparison (§2.4)

```bash
python scripts/diag_dragon_meshlet_stats.py
```

Output: `bench_meshlet_gen.csv`. Sweeps three generators (greedy, tunneling,
joint-learned) over three meshes (`stanford-bunny`, `tank`, `xyzrgb_dragon`)
and reports per-meshlet structural metrics (`#M`, `V mean`, `T mean/max`,
`T util%`, `Strips/M`, `V repl`, `Aspect med`, `Bnd%`) plus the final STRIDE
`bpv` from `encode_from_prepared`. Also prints a Markdown table to stdout
that matches the paper's Table 1 column set exactly.

Wall-clock: ≈ 3 min on the reference setup; `dragon × tunneling` is the
single largest cost. Subsequent runs hit the `utils/paradelta_cache` cache
on disk and finish in under a minute.

### Table 3 — Compression rate (8 meshes × 7 codecs)

```bash
python scripts/bench_competitors.py \
  assets/fandisk.obj assets/stanford-bunny.obj assets/horse.obj \
  assets/Monkey.obj assets/happy_buddha.obj assets/crab.obj \
  assets/tank.obj assets/xyzrgb_dragon.obj
```

Output: `bench_competitors.csv`. Columns include `bpv`, `dec_us`, `mtps`, and
`max_err`. The STRIDE row per mesh feeds both the bytes (Table 3) and the
decode throughput (Tables 4 + 5); the Draco / meshopt / Corto / DGF rows
feed the competitor columns.

Wall-clock: ≈ 8 min on the reference setup. The Numba JIT cache (`encoder/cache/`)
is created on first run and reused.

### Table 4 + Table 5 — Decode time and throughput

The numbers come from the same `bench_competitors.csv` above. For a
faster STRIDE-only sweep (no competitors, no encode) that also emits the
VRAM breakdown for Table 9:

```bash
python scripts/bench_stride_decode_sweep.py
```

Output: `bench_stride_decode_sweep.csv`. 20 warmup + 100 measured CUDA-Event-
bracketed iterations per mesh. Columns:
`kernel_us`, `mtps` (timing); `bitstream_b`, `vbuf_b`, `ibuf_b`,
`scratch_b`, `smem_per_block_b`, `smem_total_b`, `static_b`, `peak_b`
(VRAM).

### Table 6 — Encode time breakdown

```bash
python scripts/bench_stride_encode_v2.py
```

Output: `bench_stride_encode_v2.csv`. Per-mesh wall-clock for OBJ load,
dedup, partition + plan, bit-pack, and total encode. Numba JIT is warmed by
an upstream bunny encode before measurement starts.

Wall-clock: ≈ 4 min on the reference setup.

### Table 7 + Figure 7 — Bit budget breakdown

The Monkey-specific Table 7 was produced by an instrumented version of
`encode_from_prepared_v5` that reports byte counts per named section
(boundary table, connectivity stream, residual streams, headers). The
production encoder does not carry the instrumentation; the diagnostic
counterpart is `scripts/diag_v5_boundary_cost.py`, which prints the
per-meshlet bit accounting that backs §3.3 and §5.4. The instrumentation
patch lives in `legacy/diag/` (apply against the encoder, re-run, restore).

The eight-mesh accounting used in Figure 7 is produced by:

```bash
python scripts/bench_bit_budget.py
```

Output: `bench_bit_budget.csv` with per-mesh `header_bits`,
`boundary_bits`, `connectivity_bits`, `residual_bits` and their BPV
counterparts. The four buckets sum to the real `encode_from_prepared_v5`
bitstream length (bit-exact).

### Table 9 — GPU memory footprint

Same CSV as Tables 4+5. The VRAM columns (`bitstream_b`, `vbuf_b`,
`ibuf_b`, `scratch_b`, `static_b`, `peak_b`) feed Table 9 directly.

### Table 8 + 8b — Reconstruction quality and crack-free verification

```bash
python scripts/bench_recon_quality.py            # STRIDE 8 meshes + Monkey head-to-head
python scripts/verify_stride_rework.py           # cross-meshlet seam check
```

`bench_recon_quality.py` computes RMSE, symmetric mean / max Hausdorff,
and PSNR using surface-to-surface sampling (point-to-triangle, 200K
samples per direction) for STRIDE on all eight meshes (Table 8) and for
five codecs on Blender Monkey at matched 12-bit precision (Table 8b).
Outputs `bench_recon_stride.csv` and `bench_recon_competitors_monkey.csv`.
DGF and Corto rows require the third-party builds described in §2.

`verify_stride_rework.py` re-encodes each mesh and asserts that every
globally-shared vertex emits a bit-identical integer code from both
adjacent meshlets — the "zero mismatches" claim in §5.5.

## 5. Verification

The repository is shipped with the exact CSVs that produced the paper tables.
If your fresh run differs, expect:

- **Decode times (Table 4 / 5)**: ±10 % vs the reported numbers on different
  Ampere GPUs; up to 2× faster on newer SKUs (4090, 5090). Throughput on
  Ampere is roughly proportional to SM count.
- **Encode times (Table 6)**: ±20 % per stage on the same i9-12900K — JIT
  variance + I/O caching dominate the spread. Total stays within ±10 %.
- **Bits per vertex (Tables 1 + 3)**: bit-exact. If you see any drift, the
  encoder or the meshlet generator weights have changed; re-pull main.
- **Reconstruction error (Table 8)**: bit-exact. L∞ / ε is the structural
  guarantee of §5.5 and should never exceed 1.00.

The `verify_*` scripts under `scripts/` exit non-zero on any deviation
beyond a numerical tolerance.

## 6. Notes for reviewers

- The decoder is one CuPy `RawKernel` in `utils/paradelta_v5_cuda.py`.
  Reading the bitstream layout in `encoder/paradelta_v5.py` and the kernel
  source side by side reproduces §4 end to end.
- The joint-learned meshlet generator's coefficients are stored in
  `utils/meshlet_gen_joint.py::LEARNED_WEIGHTS`. The training corpus listed in
  §3.2 is disjoint from the §5 evaluation corpus.
- All STRIDE timings are decode-side only (the contribution claim); encode
  cost is reported separately in Table 6 and discussed in §5.3 paragraph 4.
- The `legacy/` directory is excluded from the artifact path — it holds
  vertex-only ablation encoders that pre-date the meshlet pipeline and is
  not referenced from any paper script.
