# STRIDE — Reproducibility Artifact

Step-by-step recipe for reproducing every measurement in the STRIDE paper
([`docs/paper_visual_computer_v6.md`](docs/paper_visual_computer_v6.md)).
Every table and figure number is generated by a script in this repository.
The eight-mesh-corpus CSVs cited in the paper are committed at the repository
root.

To read or cite the numbers, see paper Tables 1–8. To reproduce them, follow
the steps below.

---

## 1. Hardware and software

| Component | Reference setup | Notes |
|---|---|---|
| GPU         | NVIDIA RTX 3090 (Ampere SM 8.6, 24 GB) | Any CUDA 12 GPU with ≥ 8 GB VRAM works; throughput scales with SM count. |
| CPU         | Intel Core i9-12900K        | Single-threaded encode + CPU competitor timings. |
| OS          | Windows 11 (Linux also tested) | |
| CUDA        | 12.x                        | Must match the installed `cupy-cuda12x` build. |
| Python      | 3.11                        | 3.10–3.12 also tested. |

All GPU numbers (STRIDE *and* DGF) are measured locally on the reference
setup. The DGF decoder is a faithful CUDA port of the published spec —
its source lives in [`bench_cpp/dgf_decode_bench.cu`](bench_cpp/dgf_decode_bench.cu).

## 2. Install

```bash
git clone --recurse-submodules https://github.com/maletsden/meshpress.git
cd meshpress
python -m venv .venv && .venv\Scripts\activate    # Windows
# source .venv/bin/activate                       # Linux/macOS
pip install -r requirements.txt
```

`cupy-cuda12x` and `DracoPy` are the load-bearing third-party deps for the GPU
decode benchmarks (§5.3) and the Draco competitor column (§5.2). If
`pip install cupy-cuda12x` fails, consult the
[CuPy installation guide](https://docs.cupy.dev/en/stable/install.html) for
the build matching your CUDA toolkit.

Optional: build the third-party competitor binaries:

```bash
# DGF reference encoder
cd third_party/DGF-SDK && cmake -B build -S . && cmake --build build --config Release && cd ../..
# Corto encoder + bench
cd third_party/corto && cmake -B build -S . && cmake --build build --config Release && cd ../..
```

Build the C++/CUDA bench harness (DGF GPU decode, meshopt CPU decode):

```bash
cd bench_cpp
./build.sh        # Linux
build_dgf_bench.bat && build_meshopt_shim.bat    # Windows
cd ..
```

See [`bench_cpp/README.md`](bench_cpp/README.md) for build details.

## 3. Datasets

The §5.1 corpus has eight meshes. Five are auto-fetchable from public
scientific repositories (Stanford 3D Scanning, Common 3D Test Models). The
other three (Blender Monkey at high-tessellation export, Threedscans Dark
Finger Reef Crab, and the `tank` game asset) must be obtained separately.

```bash
python scripts/download_models.py --paper
```

Expected `assets/` after the fetch:

```
assets/
  fandisk.obj           # Common 3D Test Models (CC0)
  stanford-bunny.obj    # Stanford 3D Scanning Repository
  horse.obj             # Common 3D Test Models (CC0)
  Monkey.obj            # Blender Suzanne, exported with 6 levels of subdiv
  happy_buddha.obj      # Stanford 3D Scanning Repository
  crab.obj              # Threedscans "Dark Finger Reef Crab" (CC0)
  tank.obj              # Game asset (positions + indices only)
  xyzrgb_dragon.obj     # Stanford 3D Scanning Repository (xyzrgb set)
```

Vertex / triangle counts after the encoder's dedup pre-pass match paper
Table 4 (test corpus).

## 4. Reproduction recipes

Each script writes a CSV to the repository root. Re-running overwrites the
file and is idempotent.

### Table 1 — Meshlet-generator comparison (§3.2)

```bash
python scripts/diag_dragon_meshlet_stats.py
```

Sweeps three generators (greedy, tunneling, joint-learned) over three meshes
(`stanford-bunny`, `tank`, `xyzrgb_dragon`) and reports per-meshlet structural
metrics (`#M`, `V mean`, `T mean/max`, `T util%`, `Strips/M`, `V repl`,
`Aspect med`, `Bnd%`) plus the resulting STRIDE BPV.

Wall-clock: ≈ 3 min. Subsequent runs hit the `utils/paradelta_cache` on-disk
cache and finish in under a minute.

### Table 2 — End-to-end IRLP measurements (§3.5)

```bash
python scripts/smoke_irlp_endtoend.py
```

Canonical-vs-fitted bytes and encode times on the eight-mesh corpus.

### Tables 5 + 6 — Compression rate and decode throughput (8 meshes × 5 codecs)

```bash
python scripts/bench_competitors.py \
  assets/fandisk.obj assets/stanford-bunny.obj assets/horse.obj \
  assets/Monkey.obj assets/happy_buddha.obj assets/crab.obj \
  assets/tank.obj assets/xyzrgb_dragon.obj
```

Output: [`bench_competitors_q12bbox_v4.csv`](bench_competitors_q12bbox_v4.csv).
Columns include `bpv`, `dec_us`, `mtps`, `max_err`. STRIDE rows feed both
the bytes column (Table 5) and the GPU throughput column (Table 6).

For a faster STRIDE-only sweep:

```bash
python scripts/bench_stride_decode_sweep.py
```

Output: [`bench_stride_decode_sweep_q12bbox.csv`](bench_stride_decode_sweep_q12bbox.csv).
20 warm-up + 100 measured CUDA-Event-bracketed iterations per mesh.

### DGF GPU comparison (Table 6 “DGF GPU” column)

```bash
cd bench_cpp
./dgf_decode_bench        # Linux
dgf_decode_bench.exe      # Windows
cd ..
```

Output: [`bench_dgf_vs_stride.csv`](bench_dgf_vs_stride.csv). Decode-only
throughput on the same hardware as STRIDE.

### Figure 7 — Bit-budget breakdown (§5.4)

```bash
python scripts/bench_bit_budget.py
```

Per-mesh `header_bits`, `connectivity_bits`, `delta_bits`, `irlp_bits`,
`side_bits` and BPV counterparts. The four buckets sum to the encoded
bitstream length (bit-exact).

### Table 7b + recon panels (§5.5)

```bash
python scripts/bench_recon_quality.py            # STRIDE 8 meshes + Monkey head-to-head
python scripts/verify_stride_rework.py           # cross-meshlet seam check
```

`bench_recon_quality.py` computes RMSE, symmetric mean / max Hausdorff, and
PSNR using surface-to-surface sampling (point-to-triangle, 200 K samples per
direction) for STRIDE on all eight meshes (Table 7) and for five codecs on
Blender Monkey at matched 12-bit precision (Table 7b). Output:
[`bench_recon_stride_q12bbox.csv`](bench_recon_stride_q12bbox.csv).

`verify_stride_rework.py` re-encodes each mesh and asserts every globally-shared
vertex emits a bit-identical integer code from both adjacent meshlets — the
"zero mismatches" claim in §5.5.

### Encode-time breakdown

```bash
python scripts/bench_stride_encode_v2.py
```

Output: [`bench_stride_encode_v2_q12bbox.csv`](bench_stride_encode_v2_q12bbox.csv).
Per-mesh wall-clock for OBJ load, dedup, partition + plan, bit-pack, and total
encode. Numba JIT is warmed by an upstream bunny encode before measurement.

## 5. Verification

The repository ships with the exact CSVs that produced the paper tables. On a
fresh run, expect:

- **Decode times (Tables 5 + 6)**: ±10 % vs. the reported numbers on
  different Ampere GPUs; up to 2× faster on newer SKUs (4090, 5090).
- **Encode times**: ±20 % per stage on the same i9-12900K — JIT variance and
  I/O caching dominate the spread. Total stays within ±10 %.
- **Bits per vertex (Tables 1 + 5)**: bit-exact. Any drift means the encoder
  or the joint-learned partitioner weights have changed; re-pull `main`.
- **Reconstruction error**: bit-exact on the integer lattice. The L∞ vertex
  bound is Δ / 2 by construction (paper §5.5).

The `verify_*` scripts under `scripts/` exit non-zero on any deviation beyond
a numerical tolerance.

## 6. Notes for reviewers

- The decoder is one CuPy `RawKernel` in `utils/paradelta_v5_cuda.py`.
  Reading the bitstream layout in [`docs/bitstream_spec.md`](docs/bitstream_spec.md),
  the encoder source in `encoder/paradelta_v5.py`, and the kernel source side
  by side reproduces §4 end to end.
- The joint-learned meshlet generator's coefficients are stored in
  [`models/meshlet_gen_weights.json`](models/meshlet_gen_weights.json). The
  training corpus listed in §3.2 is disjoint from the §5 evaluation corpus.
- All STRIDE timings are decode-side only (the contribution claim); encode
  cost is reported separately and discussed in §3.5.
- The `legacy/` directory holds pre-STRIDE encoders (int-wavelet,
  progressive LOD, ellipsoid fitting). It is not on the paper path and is
  not referenced by any reproduction recipe in this document.
